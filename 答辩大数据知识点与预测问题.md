> 老师可能问的大数据相关问题，按概率从高到低排列，每个问题都给出简洁有力的回答话术。

## 第一梯队：几乎必问的问题

### Q1: 你这个项目的大数据技术体现在哪里？

> **回答话术：**
>
> 我的项目完整覆盖了大数据处理的六大环节。
> 第一是**数据采集**，我使用 mitmproxy 中间人代理配合 uiautomator2 自动化框架，拦截番茄小说 APP 的 API 流量，采集了大量的小说书籍、用户评论、作者信息等多维度数据。
> 第二是**数据存储**，使用 MySQL 关系型数据库，设计了 8 张表来存储结构化数据。
> 第三是**数据清洗**，对原始 JSON 数据进行了字段映射、类型转换、缺失值处理、去重等 ETL 操作。
> 第四是**数据分析**，实现了分组聚合统计、多维交叉分析等，比如按分类统计、按月度趋势分析。
> 第五是**数据挖掘**，这是我项目的核心亮点——我自主训练了一个基于 Transformer 的深度学习情感分析模型，同时实现了基于 Jaccard 相似系数的推荐算法。
> 第六是**数据可视化**，前端使用 ECharts 实现了 10 多种图表类型，并且做了一个专业的数据可视化大屏。

### Q2: 你的情感分析模型是怎么实现的？/ 讲一下你的模型

> **回答话术：**
>
> 我的情感分析模型是一个基于 Transformer Encoder 的中文情感二分类器，是我从零训练的，不是用的预训练模型。
>
> 模型结构是这样的：首先输入文本经过**字符级分词**，每个中文字符映射为一个 ID；然后通过 **Embedding 嵌入层**转换为 128 维的向量；接着加上**正弦位置编码**注入位置信息；再经过 **2 层 Transformer 编码器**，每层有 4 个注意力头，通过自注意力机制让每个字符关注到其他所有字符的信息；最后通过**均值池化**和**全连接层**输出二分类概率。
>
> 训练方面，我使用了 Adam 优化器、交叉熵损失函数、StepLR 学习率衰减策略，训练 10 个 epoch，保存验证集上准确率最高的模型。
>
> 在实际使用中，因为原始模型只有积极和消极两类，我引入了一个**置信度阈值机制**：当模型对预测结果的置信度低于 0.6 时，将其归为中性。这样就把二分类转换成了更实用的三分类。

### Q3: 什么是 Transformer？/ 自注意力机制是什么？

> **回答话术：**
>
> Transformer 是 2017 年 Google 在论文《Attention Is All You Need》中提出的深度学习架构，它的核心创新是**自注意力机制**。
>
> 简单来说，自注意力机制就是让序列中的每个位置都能**直接关注**到其他所有位置的信息。比如在"这本小说太好看了"中，当模型处理"好"这个字时，它会计算"好"和每个其他字的关联权重——"好"和"太"的关联高（程度修饰），"好"和"看"的关联高（搭配关系），这样模型就能理解"太好看"是一个正面表达。
>
> 数学上，就是通过三个矩阵 Q（查询）、K（键）、V（值），计算 softmax(QK^T/√d_k) × V。
>
> 相比 RNN 和 LSTM，Transformer 的优势是：第一，**并行计算**，不需要按顺序处理；第二，能直接建立**长距离依赖**，不存在信息遗忘问题。这也是后来 BERT、GPT 等大模型的基础架构。

### Q4: 你的数据是怎么采集的？

> **回答话术：**
>
> 我采用的是 **mitmproxy 中间人代理加上 Android UI 自动化**的采集方案。
>
> 具体流程是：首先在逍遥模拟器中运行真实的番茄小说 APP，然后启动 mitmproxy 作为 HTTPS 代理，让模拟器的所有网络流量都经过这个代理。mitmproxy 会解密 HTTPS 流量，我编写的 Addon 脚本会过滤出番茄小说服务器的 API 响应，记录下来。
>
> 同时，我用 uiautomator2 这个 Python 框架来自动化控制 APP——比如自动搜索小说、翻页浏览、进入评论区等。每次 APP 向服务器请求数据，mitmproxy 就会拦截并记录。最后将拦截到的 JSON 数据导出保存。
>
> 这种方案的优势是：获取的是 APP 端完整的 API 数据，比网页端更丰富；数据本身就是结构化的 JSON，不需要解析 HTML；而且模拟真实用户行为，反爬风险低。

### Q5: 为什么不用 Hadoop / Spark？

> **回答话术：**
>
> 这是一个很好的问题。Hadoop 和 Spark 是分布式计算框架，主要解决的是**单台机器处理不了的海量数据**的问题，通常是 TB 级甚至 PB 级的数据。
>
> 我这个项目的数据规模在 GB 级别以内，单机 MySQL 完全能够胜任存储和查询。如果为了用而用 Hadoop，反而会增加系统复杂度，这不符合工程实践中的"合适的技术解决合适的问题"原则。
>
> 但是，我的项目在**大数据处理流程和思想**上是完整的——采集、存储、清洗、分析、挖掘、可视化，每个环节都有体现。而且我用到的技术比如 Transformer 深度学习、NLP 文本分析、推荐算法，这些都是大数据分析的核心技术。
>
> 如果未来数据量增长到需要分布式处理的规模，MySQL 可以替换为 HBase 或 Hive，Python 模型可以部署到 Spark MLlib 上，整体架构是可以平滑升级的。

## 第二梯队：大概率会问的问题

### Q6: 你的推荐算法是怎么实现的？

> **回答话术：**
>
> 我实现的是**基于内容的推荐**，使用 Jaccard 相似系数来计算书籍之间的相似度。
>
> 具体来说，每本小说都有一组标签，比如"玄幻、修仙、升级、热血"。我将两本书的标签集合进行 Jaccard 计算：交集大小除以并集大小，再乘以 100 得到百分比。
>
> 在此基础上，我还加入了**分类加权**（同分类加 15 分）和**作者加权**（同作者加 10 分），使推荐更精准。最后过滤掉相似度低于 30% 的结果。
>
> 之所以选择 Jaccard 而不是协同过滤，是因为协同过滤需要大量用户行为数据，而我们平台是新建的，用户数据不够多。但书籍本身有丰富的标签信息，所以基于内容的推荐更合适。

### Q7: 前后端分离是怎么实现的？有什么优势？

> **回答话术：**
>
> 前端使用 Vue 3 框架，后端使用 Spring Boot，两者通过 RESTful API 进行通信。前端发送 HTTP 请求到后端的 `/api/` 接口，后端返回统一格式的 JSON 数据 `{code, data, msg}`。
>
> 开发环境中通过 Vite 的 proxy 配置解决跨域问题，将 `/api` 请求代理转发到后端的 8080 端口。
>
> 优势有三个：第一，前后端可以**并行开发**，提高效率；第二，前端使用 SPA 单页应用架构，用户体验更流畅，页面切换不需要刷新；第三，后端 API 是通用的，未来可以被移动端等其他客户端复用。

---

### Q8: JWT 认证是怎么工作的？

> **回答话术：**
>
> JWT 是 JSON Web Token 的缩写，是一种无状态的身份认证方案。
>
> 用户登录成功后，后端用 HMAC-SHA256 算法生成一个 Token，里面包含用户 ID 和过期时间。前端把 Token 存在 localStorage 里，之后每次请求都在 Header 中携带这个 Token。
>
> 后端收到请求后，先解码 Token 验证签名是否合法、是否过期，如果验证通过就从中提取用户 ID 继续处理业务逻辑。
>
> 它的优势是**无状态**——服务器不需要存储 Session 信息，天然适合分布式部署。我的项目中 JWT 拦截器只作用在收藏相关的接口上，其他公开的数据分析接口不需要登录。

### Q9: 你的项目架构是怎样的？（微服务相关）

> **回答话术：**
>
> 我的项目采用了**微服务**的思想。整体由三个独立的服务组成：
>
> 第一是 **Vue 前端服务**，负责页面展示和用户交互；
> 第二是 **Spring Boot 后端服务**，运行在 8080 端口，负责业务逻辑和数据接口；
> 第三是 **Python Flask 模型服务**，运行在 5001 端口，专门负责 Transformer 情感分析推理。
>
> 后端和模型服务之间通过 HTTP API 通信，这种松耦合的架构允许各服务独立开发、独立部署、独立扩展。并且我在 Java 端实现了**服务降级**——当 Python 模型服务不可用时，自动回退返回中性结果，保证系统整体可用性。

---

### Q10: 中文分词你是怎么做的？为什么不用 jieba？

> **回答话术：**
>
> 我实现了两种分词方式。
>
> 第一种用于词云展示，是**前向最大匹配算法**——从文本头部开始，尝试匹配词典中最长的词，匹配到就输出，匹配不到就缩短一个字继续尝试。词典由情感词汇和通用词汇组成。
>
> 第二种用于情感分析模型，直接采用**字符级输入**——每个中文字符就是一个 token。这是因为 Transformer 的自注意力机制可以自动学习字符之间的组合关系，相当于让模型自己学会"隐式分词"。
>
> 不用 jieba 的原因是：减少外部依赖让项目更轻量；而且对于情感分析任务，字符级输入的效果已经足够好。

## 第三梯队：可能会问的问题

### Q11: ECharts 可视化用了哪些图表？为什么选这些？

> 饼图展示占比分布（情感分布），柱状图做分类对比（类型统计），折线图展示时间趋势（月度情感变化），词云展示文本频率（热点词），河流图展示多维度时间演变（类别热度），关系图展示网络关系（相似推荐），仪表盘展示单一关键指标（好评率），散点图展示二维分布（主题聚类）。每种图表都是根据数据特点选择的最合适的可视化方式。

### Q12: 数据清洗具体做了什么？

> 字段映射（API 字段名到数据库字段名）、数据类型转换（字符串转数值）、缺失值处理（默认值填充）、数据去重（相同 ID 保留最新版本）、文本清洗（去除 HTML 标签和特殊字符）、数据校验（评分范围、时间戳合理性）。

### Q13: 如果数据量增长到百万级、千万级怎么办？

> 数据库层面可以做**分库分表**，评论表按 book_id 分表；添加 **Redis 缓存**热点查询结果；情感分析结果已经缓存在数据库中，不需要重复计算；如果需要更大规模，可以将 MySQL 替换为 **HBase**，分析任务迁移到 **Spark**。整体架构是可以平滑升级的。

### Q14: 你的模型和 BERT 有什么区别？

> BERT 是 Google 的预训练模型，有 1.1 亿参数，12 层 Transformer，768 维。我的模型只有 2 层、128 维，参数量小得多。BERT 在大规模语料上做了预训练然后微调，我的模型是从零在情感数据上直接训练的。本项目数据规模适中，轻量模型已经够用，同时也更能体现我对 Transformer 架构的理解。

### Q15: 为什么用 MySQL 而不是 MongoDB？

> 我们的数据有明确的关系结构（书籍-评论、书籍-标签、用户-收藏），需要多表联合查询和事务支持，这是关系型数据库的强项。MongoDB 更适合文档型、无固定 schema 的场景。另外 MySQL 的生态成熟，和 MyBatis-Plus 配合使用开发效率很高。

### Q16: 你怎么评估情感分析的效果？

> 模型训练过程中使用验证集计算准确率，保存最优模型。在实际应用中，通过置信度阈值机制进一步保障质量——模型不确定的样本归为中性，避免强行分类。同时，分析结果持久化到数据库，支持后续的人工抽查和校验。

### Q17: 你的项目有什么创新点？

> 第一，采集方案创新——用 mitmproxy 拦截 APP API 流量，而非传统网页爬虫；第二，自训练 Transformer 模型——不依赖预训练模型，从零构建字符级情感分类器；第三，模型服务化——Python 模型通过 Flask 服务化，Java 后端通过 HTTP 调用，实现跨语言微服务协作；第四，情感缓存机制——预测结果写回数据库，兼顾准确性和性能。

## 概念速记卡片

| 概念 | 一句话解释 |
|------|-----------|
| 大数据 5V | Volume大量、Velocity高速、Variety多样、Value价值、Veracity真实 |
| ETL | Extract提取 → Transform转换 → Load加载 |
