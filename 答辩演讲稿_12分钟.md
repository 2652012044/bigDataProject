## 第一部分：项目介绍（1 分 30 秒）

**【停留在：登录页 `/login`，先不登录】**

> 各位老师好，我今天汇报的项目是——**基于大数据技术的网络小说数据分析与情感挖掘平台**。
>
> 这个项目的背景是：随着网络文学市场的快速发展，番茄小说等平台积累了海量的小说数据和用户评论。我希望通过数据采集、存储、清洗、分析和可视化的完整技术链路，对这些数据进行多维度分析，包括类型分布、热度趋势、关键词挖掘、情感分析等，为读者提供智能化的数据洞察和推荐服务。
>
> 整个项目的技术架构分为五层：
> - **数据采集层**：使用 Python + mitmproxy 中间人代理 + uiautomator2 UI 自动化，拦截番茄小说 APP 的 API 流量
> - **数据存储层**：MySQL 8.0 数据库，设计了 8 张表存储结构化数据
> - **后端服务层**：Spring Boot 3.5 + MyBatis-Plus，提供 35 个 RESTful API 接口
> - **AI 模型层**：自训练的 Transformer 情感分析模型，通过 Flask 服务化部署
> - **前端展示层**：Vue 3 + ECharts + Element Plus，22 个页面，10 多种可视化图表
>
> 下面我通过演示来展示各个功能模块。

**【操作：输入用户名和密码，点击登录】**

## 第四部分：情感分析模块 — 核心亮点（2 分 30 秒）

**【操作：点击顶部导航 "情感分析"，进入 `/sentiment/classify`】**

### 4.1 情感分类

> 情感分析是本项目的**核心亮点模块**。
>
> 首先是**情感分类**页面。这里展示了所有用户评论的情感分布。我自主训练了一个基于 Transformer Encoder 的中文情感分析模型——不是使用 BERT 等预训练模型，而是从零训练的。
>
> 模型结构是：字符级输入 → Embedding 嵌入层（128维） → 正弦位置编码 → 2 层 Transformer 编码器（4 头注意力） → 均值池化 → 全连接二分类。原始模型输出积极/消极二分类，我通过引入 0.6 的置信度阈值，将低置信度结果归为中性，实现了三分类。
>
> 页面上方的饼图展示正面、中性、负面的**全局分布比例**。下方还有**月度情感趋势**折线图和**按分类的情感分布**，可以看到不同类型小说的用户满意度差异。

**【操作：指向饼图各部分说明比例，滚动展示月度趋势图和分类情感分布图】**

### 4.2 口碑评分

**【操作：点击左侧导航 "口碑评分"，进入 `/sentiment/reputation`】**

> **口碑评分**页面展示了小说的综合口碑排行。
>
> 排名算法是我自己设计的加权评分公式：**综合口碑指数 = 好评率 × 60% + 官方评分（归一化）× 40%**。这样既考虑了用户的真实评价，又参考了平台的官方评分，比单一指标更全面。

**【操作：展示排行表格，指出口碑指数列】**

### 4.3 热点分析

**【操作：点击左侧导航 "热点分析"，进入 `/sentiment/hotspot`】**

> **热点分析**包含两部分：上方是**热点讨论词云**，提取所有评论中的高频词汇。后端使用自实现的**前向最大匹配分词算法**对中文评论进行分词，同时配合领域停用词表过滤无意义的高频词。
>
> 下方是**热门讨论话题**排行，按评论数量排序，展示哪些书籍引发了最多的用户讨论。

**【操作：展示词云和话题排行】**

### 4.4 情感预测（实时演示）

**【操作：点击左侧导航 "情感预测"，进入 `/sentiment/predict`】**

> 这是**实时情感预测**功能，也是一个交互亮点。用户可以输入任意中文文本，系统会调用 Transformer 模型进行实时情感判断。
>
> 我现在输入一段文字来演示——

**【操作：在输入框中输入 "这本小说写得太精彩了，情节紧凑，人物性格鲜明，强烈推荐！"，点击预测按钮】**

> 可以看到模型返回了**积极**的判断，并给出了置信度分数。
>
> 再试一个负面的——

**【操作：输入 "剧情拖沓，主角太无聊了，浪费时间"，点击预测】**

> 模型准确判断为**消极**。
>
> 这个功能的技术实现是：前端 POST 请求发到 Spring Boot 后端，后端通过 HTTP 调用 Python Flask 模型服务的 `/predict` 接口，Flask 加载训练好的 Transformer 模型进行推理，返回结果。整个调用链是：**Vue → Spring Boot → Flask → PyTorch 模型**。
>
> 页面下方还展示了模型的基本信息，包括模型类型、词汇表大小、最大序列长度等参数。

**【操作：展示河流图，切换一个分类查看推荐结果】**

### 5.3 相似推荐（算法核心）

**【操作：点击左侧导航 "相似推荐"，进入 `/recommend/similar`】**

> **相似推荐**是推荐模块的核心。用户选择一本小说后，系统基于 **Jaccard 相似系数**计算标签集合的相似度。
>
> 算法实现：提取两本书的标签集合，计算交集除以并集得到基础相似度，再加上分类相同 +15 分、同作者 +10 分的加权，最后过滤掉低于 30% 的结果。
>
> 页面使用 ECharts 的**力导向关系图**展示推荐结果，中心节点是选中的书，周围节点是推荐的相似书籍，连线粗细代表相似度高低。右侧还有相似度进度条。

**【操作：选择一本书（如下拉选择第一本），等待关系图和相似度列表渲染出来，鼠标悬停关系图节点】**

# 演讲注意事项

1. **语速控制**：每个模块不要超时，如果某个页面加载慢就口头过渡
2. **重点标记**：情感分析模块和大屏是亮点，重点讲，其他模块可以快速过
3. **交互演示**：情感预测一定要现场输入文本演示，这个最能展示模型的实际效果
4. **提前准备**：
   - 确保三个服务都已启动（前端5173、后端8080、模型服务5001）
   - 提前登录一个账号
   - 提前在 `/sentiment/predict` 页面测试一下模型服务是否正常
   - 浏览器调到合适的缩放比例（建议 90% 或 100%）
   - 如果用投影仪，提前测试分辨率和显示效果
5. **应急方案**：如果模型服务 5001 没有启动，情感预测功能会返回"中性"（降级机制），其他功能不受影响

## 一、第一梯队：几乎必问（重点准备）

### Q1: 你这个项目的大数据技术体现在哪里？

**回答：**

我的项目完整覆盖了大数据处理的六大环节：

1. **数据采集**：使用 mitmproxy 中间人代理配合 uiautomator2 自动化框架，拦截番茄小说 APP 的 API 流量，采集了大量的小说书籍信息、用户评论、作者信息等多维度数据
2. **数据存储**：使用 MySQL 关系型数据库，设计了 8 张表（book、comment、author、category、tag、book_tag、web_user、user_favorite）来存储结构化数据
3. **数据清洗**（ETL）：对原始 JSON 数据进行了字段映射、类型转换、缺失值处理、去重等操作
4. **数据分析**：实现了分组聚合统计、多维交叉分析，比如按分类统计平均评分/字数/完结率、按月度分析情感趋势
5. **数据挖掘**：自训练 Transformer 情感分析模型，实现基于 Jaccard 相似系数的推荐算法，标签共现主题聚类
6. **数据可视化**：ECharts 实现了 10+ 种图表类型（饼图、柱状图、折线图、词云、河流图、关系图、仪表盘、散点图、玫瑰图），并做了专业的数据可视化大屏

### Q2: 你的情感分析模型是怎么实现的？讲一下你的模型

**回答：**

我的模型是一个基于 Transformer Encoder 的中文情感二分类器，从零训练，不依赖预训练模型。

**模型结构**：
- 输入：字符级分词，每个中文字符映射为一个整数 ID
- Embedding 嵌入层：将 ID 转换为 128 维向量
- 正弦位置编码：注入位置信息（sin/cos 函数）
- 2 层 Transformer 编码器：每层 4 个注意力头，前馈网络 256 维，LayerNorm + Dropout(0.1)
- 均值池化：对非 padding 位置取均值
- 全连接层：Linear(128, 2) + Softmax 输出二分类概率

**训练配置**：Adam 优化器，lr=1e-3，CrossEntropyLoss 损失函数，StepLR 每 3 个 epoch 学习率减半，梯度裁剪 max_norm=1.0，batch_size=32，训练 10 个 epoch，保存验证集最优模型。

**三分类转换**：原始二分类通过置信度阈值 0.6 转换——置信度 < 0.6 归为中性，高置信度积极归为 positive，高置信度消极归为 negative。

### Q3: 什么是 Transformer？自注意力机制是什么？

**回答：**

Transformer 是 2017 年 Google 在《Attention Is All You Need》论文中提出的深度学习架构，核心创新是**自注意力机制（Self-Attention）**。

**自注意力机制**：让序列中每个位置都能直接关注所有其他位置。比如处理"这本小说太好看了"时，"好"可以直接关注到"太"（程度修饰）和"看"（搭配关系），理解"太好看"是正面表达。

**数学公式**：`Attention(Q, K, V) = softmax(QK^T / √d_k) × V`
- Q（查询）、K（键）、V（值）是输入经过线性变换得到的三个矩阵
- `√d_k` 是缩放因子，防止点积过大
- 多头注意力将 Q、K、V 分成多个头并行计算，我的模型用了 4 个头

**vs RNN/LSTM 的优势**：
- 并行计算，不需要顺序处理
- 能直接建立长距离依赖，不存在梯度消失/信息遗忘
- 这也是 BERT、GPT 等大模型的基础架构

### Q5: 为什么不用 Hadoop / Spark？

**回答：**

Hadoop 和 Spark 是分布式计算框架，解决的是单机处理不了的 TB/PB 级海量数据问题。我的项目数据在 GB 级以内，单机 MySQL 完全胜任。如果为了用而用分布式框架，反而增加系统复杂度，不符合"合适的技术解决合适的问题"原则。

但我的项目在**大数据处理流程和思想**上是完整的——采集→存储→清洗→分析→挖掘→可视化，每个环节都有实现。而且用到的 Transformer 深度学习、NLP 文本分析、推荐算法，这些都是大数据分析的核心技术。

## 二、第二梯队：大概率会问

### Q6: 你的推荐算法是怎么实现的？

**回答：**

基于内容的推荐，使用 **Jaccard 相似系数**。

每本书有一组标签（如"玄幻、修仙、升级"），两本书的相似度 = |标签交集| / |标签并集| × 100。在此基础上加入加权：同分类 +15 分，同作者 +10 分，上限 100 分，低于 30 分过滤。

选择 Jaccard 而非协同过滤的原因：协同过滤需要大量用户行为数据，新平台用户少（冷启动问题），但书籍标签信息丰富，所以基于内容的推荐更合适。

### Q7: 前后端分离是怎么实现的？

**回答：**

前端 Vue 3，后端 Spring Boot，通过 RESTful API 通信。后端返回统一 JSON 格式 `{code, data, msg}`。

跨域解决：开发环境用 Vite proxy 将 `/api` 代理到 `localhost:8080`；后端也配置了 CORS 允许跨域。

前端使用 Axios 封装了请求拦截器（自动携带 JWT Token）和响应拦截器（统一处理错误）。

优势：前后端并行开发、SPA 用户体验好、API 可被其他客户端复用。

---

### Q8: JWT 认证是怎么工作的？

**回答：**

JWT（JSON Web Token）是无状态身份认证方案。

流程：用户登录 → 后端验证密码 → 用 HS256 算法生成包含 userId 和过期时间的 Token → 返回给前端 → 前端存 localStorage → 之后每次请求在 Authorization Header 携带 `Bearer <token>` → 后端拦截器解码验证 → 提取 userId 执行业务。

有效期 2 小时，过期后前端重定向登录页。只拦截 `/api/favorites/**` 路径，其他公开接口无需登录。

优势：无状态，服务器不存 Session，天然适合分布式部署。

### Q9: 项目架构是怎样的？

**回答：**

三个独立服务组成的微服务架构：
- **Vue 前端**（5173 端口）：页面展示和用户交互
- **Spring Boot 后端**（8080 端口）：业务逻辑和 35 个 API 接口
- **Python Flask 模型服务**（5001 端口）：Transformer 情感分析推理

后端和模型服务通过 HTTP API 松耦合通信。Java 端实现了**服务降级**——当 Python 服务不可用时，自动返回中性结果（置信度 0.5），保证整体可用。

后端内部采用分层架构：Controller → Service → Mapper/DAO → Model，使用 MyBatis-Plus ORM 访问 MySQL。

### Q10: 中文分词怎么做的？为什么不用 jieba？

**回答：**

两种方式：
1. **词云分词**：自实现的前向最大匹配算法——从文本头部开始匹配词典中最长的词，配合领域停用词表过滤
2. **情感模型**：字符级输入，每个字就是一个 token，让 Transformer 自注意力自动学习字符组合关系（"隐式分词"）

不用 jieba 的原因：减少外部依赖、避免分词错误传导、字符级词汇表更小模型更轻量。

### Q11: 大数据的 5V 特征是什么？你的项目体现了哪些？

### Q12: 什么是 ETL？你的项目中怎么实现的？

**回答：**

ETL = Extract（提取）→ Transform（转换）→ Load（加载）。

- **Extract**：从 mitmproxy 拦截的原始 JSON 文件中提取数据
- **Transform**：字段映射（API 字段名 → 数据库字段名）、类型转换（字符串 → 数值）、缺失值处理（默认值填充）、数据去重、文本清洗
- **Load**：清洗后的数据写入 MySQL 的 8 张表

### Q15: 为什么用 MySQL 不用 MongoDB？

### Q16: 什么是位置编码？为什么需要？

**回答：**

Transformer 没有循环结构，不像 RNN 天然有顺序信息，所以需要显式注入位置信息。

正弦位置编码使用 sin/cos 函数生成：
- `PE(pos, 2i) = sin(pos / 10000^(2i/d_model))`
- `PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))`

不同位置会得到唯一的编码向量，模型通过这些编码知道每个字符在句子中的位置。使用三角函数的好处是可以泛化到训练时没见过的长度。

---

### Q17: 什么是梯度消失？Transformer 怎么解决的？

**回答：**

梯度消失是指深层神经网络在反向传播时，梯度逐层衰减趋近于零，导致底层参数无法有效更新。RNN 在处理长序列时特别严重。

Transformer 的解决方式：
1. **自注意力**直接连接任意位置，梯度路径短
2. **残差连接**（每层输出 = 子层输出 + 子层输入），让梯度直接流过
3. **LayerNorm** 归一化，稳定梯度分布
4. 我的训练中还用了**梯度裁剪**（max_norm=1.0），防止梯度爆炸

---

### Q18: Long 精度丢失是什么问题？怎么解决的？

**回答：**

数据库的 bookId 是 19 位 BIGINT（如 7276384138653862966），超过了 JavaScript `Number.MAX_SAFE_INTEGER`（2^53-1 ≈ 16位），前端收到后精度丢失，末尾变成 0。

解决方案：Spring Boot 全局 Jackson 配置，将所有 Long 类型序列化为 String 传输，前端按字符串处理。

---

### Q19: 什么是交叉熵损失函数？

**回答：**

交叉熵衡量模型预测的概率分布与真实标签之间的差异。

对于二分类：`Loss = -[y·log(p) + (1-y)·log(1-p)]`
- y 是真实标签（0 或 1），p 是模型预测概率
- 预测越接近真实值，损失越小
- 相比均方误差，交叉熵对错误预测的惩罚更大，收敛更快

---

### Q20: MyBatis-Plus 和 MyBatis 有什么区别？

**回答：**

MyBatis-Plus 是 MyBatis 的增强版，在不改变 MyBatis 的基础上进行扩展：
- **内置 CRUD**：继承 BaseMapper 自动获得增删改查方法，不需要写 XML
- **条件构造器**：LambdaQueryWrapper 链式查询，代码更简洁
- **分页插件**：内置分页拦截器，自动处理 LIMIT/OFFSET
- 复杂 SQL 仍然可以用 XML 写（我的项目中 tag 多表关联查询就用了 XML）
