随着网络文学市场的快速发展，番茄小说等平台积累了海量小说数据和用户评论。本项目旨在通过数据爬取、存储、分析和可视化的完整技术链路，对小说数据进行多维度分析，包括类型分布、热度趋势、关键词挖掘、情感分析等，为读者提供智能化的小说推荐和评估服务。

## 二、数据采集模块

### 2.1 采集方案设计

本项目采用 **mitmproxy 中间人代理 + Android UI 自动化** 的方案，区别于传统网页爬虫：

1. 在逍遥模拟器中运行真实的番茄小说 Android APP（包名：`com.dragon.read`）
2. 使用 mitmproxy 作为 HTTPS 代理拦截 APP 与服务器之间的 API 通信
3. 通过 uiautomator2 控制 APP 进行自动化操作（搜索、翻页、点击等）
4. 拦截到的 API 响应数据以 JSON 格式存储到本地

**优势**：
- 获取的是 APP 端的完整 API 数据（比网页端更丰富）
- 不依赖 HTML 解析，数据结构稳定
- 模拟真实用户行为，反爬风险低

### 2.2 技术实现细节

```
category ──1:N──► book ──1:N──► comment
                   │
                   ├──N:1──► author
```

### 3.4 关键设计决策

1. **主键策略**：`book`、`comment`、`author` 等表使用番茄小说原始 BIGINT ID（`IdType.INPUT`），而非自增主键，保持数据一致性
2. **情感缓存**：`comment` 表的 `sentiment_label` 和 `sentiment_score` 字段用于缓存 ML 模型预测结果，避免重复计算
3. **Long 精度问题**：19 位 BIGINT ID 超出 JavaScript `Number.MAX_SAFE_INTEGER`（2^53-1），通过 Jackson 全局配置将 Long 序列化为 String 解决

### 4.6 Long 精度丢失解决方案

**问题**：数据库中 BIGINT 类型的 bookId（如 `7276384138653862966`，19 位数字）超出了 JavaScript `Number.MAX_SAFE_INTEGER`（2^53 - 1 ≈ 9007199254740991，16 位），前端收到后精度丢失，导致查询错误。

**解决**：全局 Jackson 配置，将所有 Long 类型序列化为 String：

```java
@Configuration
public class JacksonConfig {
    @Bean
    public ObjectMapper objectMapper(Jackson2ObjectMapperBuilder builder) {
        ObjectMapper mapper = builder.createXmlMapper(false).build();
        SimpleModule module = new SimpleModule();
        module.addSerializer(Long.class, ToStringSerializer.instance);
        module.addSerializer(Long.TYPE, ToStringSerializer.instance);
        mapper.registerModule(module);
        return mapper;
    }
}
```

## 五、情感分析模块（核心亮点）

### 5.1 模型架构 — Transformer Encoder

本项目自主训练了一个基于 Transformer Encoder 的中文情感二分类模型，而非使用预训练模型（如 BERT）。

#### 5.1.1 模型结构

```
输入文本（字符级分词）
    │
    ▼
Embedding(vocab_size, 128, padding_idx=0)    ← 字符嵌入层
    │
    ▼
PositionalEncoding(d_model=128, max_len=512) ← 正弦位置编码
    │                                           sin(pos / 10000^(2i/d))
    ▼
TransformerEncoderLayer × 2                   ← 2 层 Transformer 编码器
    │  • 多头自注意力(4 heads)
    │  • 前馈网络(256 维)
    │  • LayerNorm + Dropout(0.1)
    ▼
Mean Pooling (masked)                         ← 对非 padding 位置取均值
    │
    ▼
Dropout(0.1)
    │
    ▼
Linear(128, 2) → Softmax                     ← 二分类输出
    │
    ▼
[P(消极), P(积极)]                            ← 概率分布
```

#### 5.1.2 关键超参数

| 参数 | 值 | 说明 |
|------|------|------|
| d_model | 128 | 隐藏层维度 |
| nhead | 4 | 多头注意力头数 |
| num_layers | 2 | Transformer 编码层数 |
| dim_feedforward | 256 | 前馈网络维度 |
| max_len | 256 | 最大序列长度 |
| dropout | 0.1 | Dropout 率 |
| vocab_size | ~数千 | 字符级词汇表大小 |
| num_classes | 2 | 0=消极, 1=积极 |

#### 5.1.3 字符级分词（非分词器）

与常见的基于分词的 NLP 模型不同，本模型采用 **字符级（char-level）** 输入：

```python
class Vocabulary:
    # 每个中文字符/符号映射为一个整数 ID
    # 特殊标记：<PAD>=0, <UNK>=1
    def encode(self, text):
        return [self.word2idx.get(ch, self.word2idx['<UNK>']) for ch in text]
```

**优势**：
- 不依赖外部分词工具（如 jieba）
- 避免分词错误对情感判断的影响
- 词汇表更小，模型更轻量

#### 5.1.4 自注意力机制（Self-Attention）

Transformer 的核心是自注意力机制，允许模型关注输入序列中任意位置的信息：

```
Attention(Q, K, V) = softmax(QK^T / √d_k) × V

其中：
- Q, K, V 分别是查询、键、值矩阵
- d_k = d_model / nhead = 128 / 4 = 32
- 多头注意力将 Q, K, V 分成 4 个头并行计算
```

#### 5.1.5 位置编码（Positional Encoding）

由于 Transformer 没有循环结构，需要显式注入位置信息：

```
PE(pos, 2i)   = sin(pos / 10000^(2i/d_model))
PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))
```

#### 5.2.2 训练流程

```
1. 从训练集构建字符级词汇表 → vocab.json
2. 加载训练集和验证集（制表符分隔：文本\t标签）
3. 每个 epoch:
   a. 训练：前向传播 → 计算损失 → 反向传播 → 梯度裁剪 → 参数更新
   b. 验证：在验证集上计算准确率
   c. 保存最优模型（按验证集准确率）→ best_model.pth
4. 学习率每 3 个 epoch 衰减 50%
```

```python
    elif pred == 1:              # 高置信度 + 积极

    else:                        # 高置信度 + 消极
```

## 九、答辩常见问题预备

### Q1: 为什么选择 mitmproxy 而不是 Scrapy 等传统爬虫？
**A**: 番茄小说是移动端 APP，没有对应的网页版可供爬取。mitmproxy 可以拦截 APP 的真实 API 请求，获取结构化的 JSON 数据，比解析 HTML 更稳定。同时结合 uiautomator2 进行 UI 自动化，模拟真实用户操作，规避反爬检测。

### Q2: Transformer 模型与 BERT 的区别是什么？
**A**: 本项目的模型只使用了 Transformer 的 Encoder 部分（类似 BERT），但区别在于：(1) 从零训练，不使用预训练权重；(2) 字符级输入，不依赖分词器；(3) 模型更轻量（2 层、128 维），适合本项目规模的数据。BERT 是大规模预训练模型（12 层、768 维），参数量远大于本模型。

### Q3: 如何处理情感分析中的"中性"类别？
**A**: 原始模型是二分类（积极/消极）。通过引入置信度阈值（0.6），当模型对预测结果的置信度低于阈值时，将其归为"中性"。这比直接训练三分类模型更灵活，因为中性评论通常是模型"不确定"的结果。

### Q4: 为什么用字符级而不是词级分词？
**A**: 字符级输入的优势：(1) 不依赖外部分词工具（如 jieba），减少依赖；(2) 避免分词错误传导到情感判断；(3) 词汇表更小，模型更轻量；(4) Transformer 的自注意力机制可以自动学习字符之间的组合关系，一定程度上"隐式分词"。

### Q5: 前后端是如何解决跨域问题的？
**A**: 开发环境通过 Vite 的 proxy 配置将 `/api` 请求转发到 `http://localhost:8080`。后端 `WebMvcConfiguration.java` 也配置了 CORS，允许所有来源和方法的跨域请求。

### Q6: 推荐算法的原理是什么？
**A**: 相似推荐基于 Jaccard 相似系数计算标签集合的相似度（|交集|/|并集|×100），再加上分类奖励（+15 分）和作者奖励（+10 分）。低于 30% 相似度的结果被过滤，确保推荐质量。

### Q7: 项目中的大数据技术体现在哪里？
**A**: (1) 大规模数据采集与处理（海量小说数据和评论）；(2) 数据清洗与结构化存储；(3) 多维度数据分析与统计；(4) 深度学习模型训练与推理；(5) 数据可视化大屏展示。虽然未使用 Hadoop/Spark 等分布式框架，但项目完整覆盖了数据采集→存储→分析→展示的大数据分析全流程。

### Q8: JWT 令牌过期后怎么处理？
**A**: JWT 有效期为 2 小时。过期后前端请求会收到 401 状态码，前端路由守卫检测到未认证状态后，自动重定向到登录页面，用户重新登录获取新令牌。

### Q9: 情感模型服务宕机会影响系统吗？
**A**: 不会。`SentimentModelClient` 内置了 fallback 机制，当 Python 模型服务不可用时，自动返回"中性"结果（置信度 0.5），确保系统整体可用性。同时已分析过的评论结果缓存在数据库中，不依赖实时模型调用。
