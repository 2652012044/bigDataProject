## 二、数据采集技术

### 2.1 网络爬虫与数据采集

**什么是网络爬虫？**
网络爬虫（Web Crawler）是自动化程序，用于从互联网上获取数据。传统爬虫通过 HTTP 请求获取网页 HTML，然后解析提取数据。

**本项目的采集方案——API 流量拦截**

本项目采用更高级的方案：mitmproxy 中间人代理 + Android UI 自动化。

```
普通爬虫：  浏览器 → 请求网页 HTML → 解析 HTML → 提取数据
本项目：    APP → API请求 → mitmproxy拦截 → 直接获取JSON → 结构化数据
```

### 2.2 mitmproxy 中间人代理原理

**MITM（Man-In-The-Middle）中间人攻击/代理**：

```
正常通信：   APP ────────────────────► 番茄小说服务器
             APP ◄──────────────────── 番茄小说服务器

加入代理后： APP ──► mitmproxy ──► 番茄小说服务器
             APP ◄── mitmproxy ◄── 番茄小说服务器
                        │
                        ▼
                   记录所有数据
```

工作流程：
1. mitmproxy 作为 HTTPS 代理运行，生成自签名 CA 证书
2. 模拟器安装并信任该 CA 证书
3. 模拟器的网络流量通过 mitmproxy 转发
4. mitmproxy 解密 HTTPS 流量，记录请求和响应内容
5. 我们的 Addon 脚本过滤 `fqnovel.com` 域名的流量，提取 JSON 数据

**为什么不用传统爬虫？**
- 番茄小说是移动端 APP，没有完整的网页版
- APP 端 API 返回的数据比网页版更丰富、更结构化
- API 返回的是 JSON 格式，不需要解析 HTML
- 配合 UI 自动化模拟真实用户行为，被反爬检测的风险更低

### 2.3 UI 自动化技术

**uiautomator2** 是 Android UI 自动化测试框架：
- 通过 ADB（Android Debug Bridge）连接模拟器
- 可以模拟用户的所有操作：点击、滑动、输入、截图
- 通过 XPath/resourceId 定位 UI 元素

```python
# 示例：自动搜索小说
device = u2.connect('127.0.0.1:21503')  # 连接模拟器
app = device.session('com.dragon.read')   # 打开番茄小说
app(resourceId='search_box').click()      # 点击搜索框
app.send_keys('斗破苍穹')                 # 输入搜索关键词
app(text='搜索').click()                  # 点击搜索
# 此时 mitmproxy 已拦截到搜索相关的 API 请求
```

## 三、数据存储技术

### 3.1 关系型数据库 MySQL

**为什么选择 MySQL？**
- 数据之间存在明确的关系（书籍↔评论、书籍↔标签、用户↔收藏）
- 需要复杂的联合查询和聚合统计
- 事务支持保障数据一致性
- 成熟稳定，社区活跃

**核心表设计理念：**
- 使用原始平台 ID 作为主键（`IdType.INPUT`），保持数据溯源性
- 多对多关系通过中间表实现（`book_tag`、`user_favorite`）
- 评论表预留情感分析结果字段（`sentiment_label`、`sentiment_score`），实现计算结果持久化缓存

## 四、数据清洗与预处理（ETL）

### 4.1 什么是 ETL？

ETL = Extract（提取）+ Transform（转换）+ Load（加载）

```
Extract:   从 JSON 文件中提取原始数据
Transform: 数据清洗、格式转换、去重、过滤
Load:      导入 MySQL 数据库
```

### 4.2 本项目的数据清洗流程

```
原始 JSON 数据
    │
    ├── 字段映射：API字段名 → 数据库字段名
    │   (如 book_name → bookName, word_count → wordNumber)
    │
    ├── 数据类型转换
    │   (字符串 → 数值, 时间戳 → 日期格式)
    │
    ├── 缺失值处理
    │   (评分为空 → 默认0, 字数为空 → 默认0)
    │
    ├── 数据去重
    │   (相同 book_id 的书籍只保留最新版本)
    │
    ├── 文本清洗
    │   (评论内容去除HTML标签、特殊字符)
    │
    └── 数据校验
        (评分范围0-10, 字数>=0, 时间戳合理性)
```

### 4.3 评论文本预处理

针对情感分析的文本预处理：

```
原始评论 → 去除HTML标签 → 去除特殊字符 → 去除空白 → 长度截断(256字符)
```

| **交叉分析** | 多维度联合统计 | 分类×情感交叉分布 |

### 5.2 文本分析

#### 5.2.1 中文分词

本项目使用自实现的 **前向最大匹配算法（Forward Maximum Matching, FMM）**：

```
算法原理：
  输入：  "这本小说太好看了"
  词典：  {"小说", "好看", "太", ...}
  最大词长：4

  扫描过程：
  位置0: "这本小说" 不在词典 → "这本小" 不在 → "这本" 不在 → "这"(单字)
  位置1: "本小说太" 不在 → "本小说" 不在 → "本小" 不在 → "本"
  位置2: "小说太好" 不在 → "小说太" 不在 → "小说" 在词典! ✓
  位置4: "太好看了" 不在 → "太好看" 不在 → "太好" 不在 → "太"
  位置5: "好看了" 不在 → "好看" 在词典! ✓
  位置7: "了" 单字

  结果：  ["这", "本", "小说", "太", "好看", "了"]
```

**不使用 jieba 等外部分词库的原因**：
- 减少外部依赖，项目更轻量
- 情感分析场景下，词典可以专门针对小说评论领域优化
- 字符级 Transformer 模型不需要分词，分词仅用于词云展示

#### 5.2.2 词频统计与词云

```
全部评论文本
    → 分词
    → 去停用词（50+ 领域停用词）
    → 统计词频
    → 取 Top 50 高频词
    → 生成词云可视化
```

停用词示例：`作者、小说、主角、剧情、评论、推荐、感觉` 等无情感意义的高频结构词

#### 5.2.3 关键词提取与主题分析

```
标签提取：   每本小说的 tags 字段按逗号分割
标签聚合：   统计每个标签出现的频次
共现分析：   计算标签之间的共现频率
主题聚类：   基于标签共现关系进行聚类，生成散点图坐标
```

### 5.3 推荐算法

#### 5.3.1 基于内容的推荐（Content-Based Filtering）

本项目的相似推荐属于 **基于内容的推荐**：根据物品（小说）本身的属性（标签、分类、作者）计算相似度。

```
与之对比的推荐方法：
- 协同过滤：基于用户行为（"喜欢A的人也喜欢B"）
- 混合推荐：结合多种方法
本项目选择基于内容：数据中有丰富的标签信息，适合内容相似度计算
```

#### 5.3.2 Jaccard 相似系数

```
Jaccard(A, B) = |A ∩ B| / |A ∪ B|

示例：
  书A标签 = {玄幻, 修仙, 升级, 热血}
  书B标签 = {玄幻, 修仙, 冒险, 战斗}

  交集 = {玄幻, 修仙}，大小 = 2
  并集 = {玄幻, 修仙, 升级, 热血, 冒险, 战斗}，大小 = 6

  Jaccard = 2/6 = 0.333 → 33.3%

  + 同分类加 15% → 48.3%
  + 同作者加 10%（如果适用）

  最终相似度 = 48.3%（≥30% 阈值，保留推荐）
```

**为什么选择 Jaccard 而不是余弦相似度？**
- 标签是离散的二值特征（有/无），不是连续向量
- Jaccard 专门针对集合相似度设计，更直观
- 计算简单高效

## 六、深度学习与 NLP 技术

### 6.1 深度学习基础概念

| 概念 | 解释 | 本项目应用 |
|------|------|-----------|
| **神经网络** | 模拟人脑神经元连接的数学模型 | Transformer 中的前馈网络 |
| **嵌入层(Embedding)** | 将离散的字符/词映射为连续向量 | 字符 ID → 128 维向量 |
| **注意力机制(Attention)** | 让模型关注输入中最重要的部分 | 多头自注意力，4 个头 |
| **损失函数(Loss)** | 衡量模型预测与真实值的差距 | 交叉熵损失 CrossEntropyLoss |
| **优化器(Optimizer)** | 根据梯度更新模型参数 | Adam 优化器 |
| **过拟合(Overfitting)** | 模型在训练集上表现好但泛化差 | Dropout(0.1) 防止过拟合 |

### 6.2 Transformer 架构详解

Transformer 是 2017 年 Google 提出的深度学习架构（论文：*Attention Is All You Need*），是当前 NLP 领域的主流模型。

#### 6.2.1 核心思想——自注意力机制

传统 RNN/LSTM 按顺序处理文本，Transformer 通过注意力机制 **并行** 处理所有位置：

```
输入："这 本 小 说 太 好 看 了"

自注意力计算：
  对于"好"字，模型会计算它与每个字的关联权重：
    "好" → "太" 权重高（程度修饰关系）
    "好" → "看" 权重高（搭配关系"好看"）
    "好" → "这" 权重低（距离远，关联弱）
    "好" → "了" 权重中（语气助词，有一定关联）

  这样模型能理解"太好看"是一个正面表达
```

#### 6.2.2 数学公式

```
注意力计算：
  Q = X × W_Q    (查询矩阵)
  K = X × W_K    (键矩阵)
  V = X × W_V    (值矩阵)

  Attention(Q, K, V) = softmax(Q × K^T / √d_k) × V

  其中 d_k = d_model / nhead = 128 / 4 = 32

多头注意力：
  将 Q, K, V 分成 4 个头（每头 32 维）
  各头独立计算注意力
  拼接结果并线性变换
  → 不同头可以关注不同类型的模式
```

#### 6.2.3 位置编码

Transformer 没有循环结构，需要注入位置信息：

```
PE(pos, 2i)   = sin(pos / 10000^(2i/128))
PE(pos, 2i+1) = cos(pos / 10000^(2i/128))

例如：
  位置0的编码 = [sin(0), cos(0), sin(0), cos(0), ...]
  位置1的编码 = [sin(1/10000^0), cos(1/10000^0), sin(1/10000^(2/128)), ...]

每个位置得到唯一的128维编码向量，与字符嵌入相加
```

#### 6.2.4 本项目模型结构

```
输入: "这本小说真不错" → 字符ID: [45, 23, 67, 89, 12, 8, 56]
  │
  ▼
Embedding: [45,23,67,...] → 7个128维向量  (查表操作)
  │
  ▼
+ PositionalEncoding: 加入位置信息     (sin/cos编码)
  │
  ▼
TransformerEncoderLayer #1:
  │ MultiHeadAttention(4heads) → 每个字关注其他所有字
  │ + Add & LayerNorm
  │ FeedForward(128→256→128) → 非线性变换
  │ + Add & LayerNorm
  ▼
TransformerEncoderLayer #2: (同上结构)
  │
  ▼
Mean Pooling: 7个128维向量 → 1个128维向量  (取均值)
  │                                         (忽略padding位置)
  ▼
Dropout(0.1): 随机丢弃10%的神经元          (防过拟合)
  │
  ▼
Linear(128→2): 128维 → [消极概率, 积极概率]
  │
  ▼
Softmax: [0.15, 0.85] → 积极(85%置信度)
```

### 6.3 模型训练过程

```
训练数据: text<TAB>label 格式
  "这本书太好看了\t1"        (1=积极)
  "垃圾小说浪费时间\t0"      (0=消极)

训练循环（10个Epoch）:
  Epoch 1: lr=1e-3
    → 遍历所有训练样本(batch_size=32)
    → 前向传播：文本 → 模型 → 预测概率
    → 计算损失：CrossEntropy(预测, 真实标签)
    → 反向传播：计算每个参数的梯度
    → 梯度裁剪：防止梯度爆炸(max_norm=1.0)
    → 参数更新：Adam优化器更新权重
    → 验证集评估：计算准确率
    → 如果是最优准确率 → 保存模型

  Epoch 2: lr=1e-3 (继续)
  Epoch 3: lr=1e-3 (继续)
  Epoch 4: lr=5e-4 (StepLR衰减: ×0.5)
  Epoch 7: lr=2.5e-4 (再次衰减)
  Epoch 10: lr=1.25e-4 (再次衰减)

最终输出: best_model.pth (最优模型权重)
```

### 6.4 情感分析技术

#### 6.4.1 什么是情感分析？

情感分析（Sentiment Analysis）是 NLP 的重要任务，目标是识别文本中表达的情感倾向：
- **正面（Positive）**：好看、推荐、精彩
- **中性（Neutral）**：一般、还行、普通
- **负面（Negative）**：垃圾、无聊、差评

#### 6.4.2 本项目的情感分析方案

```
方案一：基于字典的情感分析（Java端，备用）
  ┌─────────┐   ┌──────────┐   ┌──────────┐   ┌─────────┐
  │ 输入文本 │──►│ 滑动窗口 │──►│ 匹配词典 │──►│ 计算得分│
  └─────────┘   │ 逐字扫描 │   │正面词+1  │   │>0.3=正面│
                └──────────┘   │负面词-1  │   │<-0.3=负面│
                               │否定词翻转│   │其他=中性│
                               │程度词放大│   └─────────┘
                               └──────────┘

方案二：Transformer深度学习模型（Python端，主力）
  ┌─────────┐   ┌──────────┐   ┌────────────┐   ┌─────────┐
  │ 输入文本 │──►│ 字符编码 │──►│ Transformer│──►│ 三分类  │
  └─────────┘   │ + 位置编码│   │ 自注意力   │   │正面/中性│
                └──────────┘   │ 2层编码器  │   │/负面    │
                               └────────────┘   └─────────┘
```

#### 6.4.3 置信度阈值机制

```
模型输出: [P(消极), P(积极)]

情况1: [0.05, 0.95] → 积极概率95% > 60% → 正面(positive)
情况2: [0.92, 0.08] → 消极概率92% > 60% → 负面(negative)
情况3: [0.45, 0.55] → 最大概率55% < 60% → 中性(neutral)

阈值设为0.6的原因：
  模型只训练了二分类，但现实中存在中性评论
  当模型"不确定"时（概率接近50:50），说明文本情感不明显
  将这类不确定的结果归为中性，是合理的工程处理
```

## 七、数据可视化技术

### 7.1 可视化理论

**数据可视化**是将抽象数据转换为直观图形，帮助用户理解数据模式和趋势。

本项目使用的可视化类型及其适用场景：

| 折线图 | 时间趋势 | 情感趋势（月度好评率变化） |

## 八、后端架构技术

### 8.1 RESTful API 设计

REST（Representational State Transfer）是一种 API 设计风格：

```
资源定位：  /api/book/{id}          — 通过 URL 定位资源
HTTP 方法：  GET=查询, POST=创建, PUT=更新, DELETE=删除
无状态：    每个请求包含所有必要信息（通过JWT令牌）
统一响应：  { code: 0, data: {...}, msg: "success" }
```

### 8.2 MyBatis-Plus ORM

ORM（Object-Relational Mapping）将数据库表映射为 Java 对象：

```java
// 实体类 → 数据库表
@TableName("book")
public class Book {
    @TableId(value = "book_id", type = IdType.INPUT)
    private Long bookId;       // → book_id 列
    private String bookName;   // → book_name 列 (驼峰→下划线自动转换)
    private BigDecimal score;  // → score 列
}

// 查询 → SQL
bookMapper.selectList(
    new LambdaQueryWrapper<Book>()
        .eq(Book::getCategory, "玄幻")
        .gt(Book::getScore, 8.0)
        .orderByDesc(Book::getReadCount)
        .last("LIMIT 20")
);
// 等价SQL: SELECT * FROM book WHERE category='玄幻' AND score>8.0
//          ORDER BY read_count DESC LIMIT 20
```

### 8.3 JWT 认证机制

JWT（JSON Web Token）是一种无状态的身份认证方案：

```
结构：Header.Payload.Signature

Header:  {"alg": "HS256", "typ": "JWT"}  → Base64编码
Payload: {"userId": 123, "exp": 过期时间} → Base64编码
Signature: HMAC-SHA256(header + "." + payload, 密钥)

验证流程：
  1. 收到Token后Base64解码Header和Payload
  2. 用同一密钥重新计算签名
  3. 对比签名是否一致 → 确保数据未被篡改
  4. 检查exp字段是否过期
```

**优势**：无需服务端存储 Session，适合分布式部署

### 8.4 微服务通信

后端 Spring Boot 与 Python Flask 情感模型之间通过 **HTTP RESTful API** 通信：

```
Spring Boot (Java)                    Flask (Python)
     │                                     │
     │  POST /predict                      │
     │  {"text": "好看"}                   │
     │ ──────────────────────────────────►  │
     │                                     │ ← Transformer推理
     │  {"label":"positive","conf":0.85}   │
     │ ◄──────────────────────────────────  │
     │                                     │
```

这是一种 **微服务架构** 的实践：
- 各服务独立部署、独立扩展
- 通过 HTTP API 松耦合通信
- 服务降级：Python 服务不可用时 Java 端自动 fallback

### 9.1 Vue 3 响应式原理

Vue 3 使用 **Proxy** 实现数据响应式：


## 十、答辩高频问题速查

### Q: 你的项目用到了哪些大数据技术？
**A**: 数据采集（mitmproxy API拦截 + uiautomator2 自动化）、数据存储（MySQL 关系型数据库）、数据清洗（ETL 流程）、数据分析（统计分析、文本分析、关键词提取）、数据挖掘（Transformer 深度学习情感分析、Jaccard 相似度推荐）、数据可视化（ECharts 10+ 种图表 + 数据大屏）。

### Q: Transformer 和 RNN/LSTM 的区别？
**A**: RNN/LSTM 按序列顺序处理，有长距离依赖问题；Transformer 通过自注意力机制并行处理所有位置，能直接建立任意两个位置之间的关联，计算效率更高，效果更好。这也是 BERT、GPT 等现代大模型的基础架构。

### Q: 为什么不直接用 BERT？
**A**: BERT 是大规模预训练模型（1.1亿参数），训练和部署需要较多计算资源。本项目数据规模适中，自训练的轻量 Transformer（2层、128维，约几十万参数）已经能满足需求，同时也能更好地理解和解释模型原理。

### Q: 数据量有多大？算大数据吗？
**A**: 本项目采集了番茄小说平台的书籍、评论、作者等多维度数据。"大数据"不仅指数据量大，更强调处理非结构化数据（文本评论）的能力和完整的数据处理流程。本项目完整覆盖了大数据处理的六大环节：采集→存储→清洗→分析→挖掘→可视化。

### Q: 情感分析的准确率怎么样？
**A**: 模型在验证集上取最优准确率保存。通过置信度阈值机制（<0.6 归为中性），进一步提高了实际应用中的判断质量。同时所有预测结果缓存在数据库中，支持后续人工校验和模型迭代优化。

### Q: 推荐算法为什么选 Jaccard 而不是协同过滤？
**A**: 协同过滤需要大量用户行为数据（如用户评分矩阵），本项目的用户系统是新建的，缺少足够的用户行为数据。而书籍本身有丰富的标签信息，基于 Jaccard 相似度的内容推荐更适合当前数据特点。

### Q: 前后端分离的优势是什么？
**A**: (1) 前后端可以并行开发，提高效率；(2) 前端使用 SPA 架构，用户体验更流畅；(3) 后端 API 可以被多个客户端复用（Web、移动端等）；(4) 各自独立部署和扩展。

### Q: 为什么情感模型用 Python 而不是 Java？
**A**: Python 是深度学习领域的标准语言，PyTorch/TensorFlow 等框架对 Python 支持最好。通过 Flask 服务化后，Java 后端通过 HTTP 调用 Python 模型，实现了跨语言协作，这也是微服务架构的典型实践。
